# CV Web App

A Flutter web monorepo containing two applications: a **public-facing CV/portfolio website** and an **admin dashboard** for managing content. Both apps share a common codebase of core packages and connect to Firebase as the backend.

The project follows **Clean Architecture** with modular feature packages, uses **Dart workspaces** with **Melos** for monorepo management, and supports **dev/prod environment flavors** backed by separate Firebase projects.

## Getting Started

```bash
# 1. Clone the repository
git clone <repo-url>
cd cv-web-app

# 2. Install the pinned Flutter version via FVM
fvm install

# 3. Resolve all workspace dependencies (Dart workspaces handle linking automatically)
fvm flutter pub get

# 4. Run code generation across all packages that need it
fvm dart run melos build

# 5. Generate localization files
fvm dart run melos l10n

# 6. Run the CV app in dev mode
fvm dart run melos run run:cv:dev
```

If your shell has FVM shims configured (so `flutter` already points to the FVM-managed SDK), you can drop the `fvm` prefix from all commands above.

## Architecture Overview

The project implements **Clean Architecture** with a clear dependency direction:

```
Feature Packages (presentation) --> core/domain <-- core/data
                                        ^
                                        |
                                    core/shared (utilities, config, DI)
```

### Layers

| Layer | Package | Contains |
|-------|---------|----------|
| **Domain** | `packages/core/domain` | Entities, repository interfaces (abstract classes), use cases. No Flutter/Firebase dependencies. |
| **Data** | `packages/core/data` | Repository implementations, Firebase datasources, data models (DTOs). Depends on domain. |
| **Shared** | `packages/core/shared` | Cross-cutting concerns: `IAppConfig` interface, `AppTheme`, localization, `AppBlocObserver`, `AppRouteObserver`, DI setup, generated colors. |
| **Features** | `packages/features/*` | Self-contained feature modules. Each owns its presentation layer (pages, cubits/blocs), its own DI module, and its own micro-router. |
| **Apps** | `apps/*` | Thin shell applications. Wire up DI, Firebase, routing, and theme. Contain environment configs and entry points. |

### Key Libraries

| Concern | Library |
|---------|---------|
| **Dependency Injection** | `get_it` + `injectable` (with `@dev`/`@prod` environment annotations) |
| **State Management** | `flutter_bloc` (BLoC / Cubit pattern) |
| **Routing** | `auto_route` v11 with generated type-safe routes |
| **Immutable Models** (planned) | `freezed` + `freezed_annotation` -- not yet added to any package |
| **JSON Serialization** (planned) | `json_annotation` + `json_serializable` (via build_runner) -- not yet added to any package |
| **Functional Error Handling** (planned) | `dartz` (`Either` type) -- not yet added to any package |
| **Asset Generation** | `flutter_gen` (color constants from XML) |
| **Linting** | `very_good_analysis` |
| **Firebase** | `firebase_core`, `firebase_analytics` |

### Dependency Injection Flow

Each package that registers dependencies exposes a `configure*Dependencies(GetIt, {environment})` function. The app's DI entry point (`apps/cv_app/lib/di/injection.dart`) orchestrates them:

```dart
Future<void> configureDependencies({required String environment}) async {
  // 1. Shared dependencies first
  await shared.configureSharedDependencies(getIt, environment: environment);

  // 2. Feature modules
  await cv_content.configureCvContentDependencies(getIt, environment: environment);

  // 3. App-specific dependencies (IAppConfig registered per environment)
  getIt.init(environment: environment);

  // 4. Register IFirebaseConfig by casting the IAppConfig instance
  getIt.registerSingleton<IFirebaseConfig>(
    getIt<IAppConfig>() as IFirebaseConfig,
  );
}
```

When adding a new package with injectable dependencies, add its `configure*Dependencies` call here.

## Environment and Flavor Setup

The project uses two Firebase projects and two app entry points to separate dev and prod environments.

### How It Works

1. **Entry points**: `main_dev.dart` calls `bootstrap(environment: Environment.dev)`, `main_prod.dart` calls `bootstrap(environment: Environment.prod)`. The `Environment.dev` and `Environment.prod` constants come from the `injectable` package.

2. **IAppConfig interface** (`packages/core/shared/lib/config/app_config.dart`): Defines the contract for environment-specific configuration:
   ```dart
   abstract class IAppConfig {
     String get environment;
     String get appName;
     bool get isLogBlocChanges;
     bool get isLogBlocErrors;
   }
   ```
   > `IFirebaseConfig` is a separate interface for Firebase options (`FirebaseOptions getFirebaseOptions()`), located in `config/firebase_config.dart`.

3. **Environment implementations** (`apps/cv_app/lib/config/dev/` and `prod/`):
   - `AppConfigDev` is annotated `@dev` and `@Injectable(as: IAppConfig)` -- registered only when `environment == "dev"`.
   - `AppConfigProd` is annotated `@prod` and `@Injectable(as: IAppConfig)` -- registered only when `environment == "prod"`.
   - Each class implements both `IAppConfig` and `IFirebaseConfig`, returning its own `FirebaseOptions` via `getFirebaseOptions()`, pointing to the appropriate Firebase project.

4. **Firebase projects**:
   - Dev: `cv-web-app-dev` (Firebase options in `config/dev/firebase_options_dev.dart`)
   - Prod: `cv-web-app-prod` (Firebase options in `config/prod/firebase_options_prod.dart`)

### Running a Specific Environment

```bash
# Dev (from project root)
fvm dart run melos run run:cv:dev

# Or directly from apps/cv_app:
flutter run -d chrome -t lib/main_dev.dart

# Prod:
flutter run -d chrome -t lib/main_prod.dart
```

## Code Generation

This project uses `build_runner` extensively. The following file patterns are generated and should not be edited by hand:

| Pattern | Generator | Trigger |
|---------|-----------|---------|
| `*.freezed.dart` | Freezed | Annotating a class with `@freezed` |
| `*.g.dart` | json_serializable / injectable_generator | `@JsonSerializable`, `@InjectableInit` |
| `*.gr.dart` | auto_route_generator | `@AutoRouterConfig`, `@RoutePage()` |
| `lib/gen/colors.gen.dart` | flutter_gen | `assets/colors/colors.xml` in the `shared` package |
| `lib/l10n/app_localizations*.dart` | flutter gen-l10n | ARB files in `packages/core/shared/l10n/` |

### Running Code Generation

```bash
# All packages (sequential, recommended for clean builds)
melos build

# All packages (parallel, faster for incremental builds)
melos build:concurrent

# Single package (run from within that package directory)
flutter pub run build_runner build --delete-conflicting-outputs
```

After modifying any annotated class (`@freezed`, `@Injectable`, `@RoutePage`, `@AutoRouterConfig`, `@JsonSerializable`), re-run code generation for the affected package.

## Routing with AutoRoute

The project uses **AutoRoute v11** with a micro-router pattern where each feature package defines its own router, and the app-level router aggregates them.

### How It Works

1. **Feature micro-router** (e.g., `packages/features/cv_content/lib/router/cv_content_router.dart`):
   ```dart
   @AutoRouterConfig(replaceInRouteName: 'Page,Route')
   class CvContentRouter extends RootStackRouter {
     @override
     List<AutoRoute> get routes => [
       AutoRoute(page: HomeRoute.page),
     ];
   }
   ```
   This generates `cv_content_router.gr.dart` containing `HomeRoute`, `HomeRouteArgs`, etc.

2. **Page annotation** (e.g., `home_page.dart`):
   ```dart
   @RoutePage()
   class HomePage extends StatefulWidget { ... }
   ```

3. **App-level router** (`apps/cv_app/lib/router/router.dart`):
   ```dart
   @AutoRouterConfig(replaceInRouteName: 'Page,Route')
   class AppRouter extends RootStackRouter {
     @override
     List<AutoRoute> get routes => [
       AutoRoute(page: HomeRoute.page, initial: true),
     ];
   }
   ```
   The app router imports the generated route classes from the feature package and composes them into the final route tree.

4. **Integration in App widget** (`apps/cv_app/lib/app/app.dart`):
   ```dart
   MaterialApp.router(
     routerDelegate: _appRouter.delegate(
       navigatorObservers: () => [AppRouteObserver()],
     ),
     routeInformationParser: _appRouter.defaultRouteParser(),
   )
   ```

### Route Observer

`AppRouteObserver` (in `packages/core/shared`) logs all navigation events (push, pop, replace, remove) for debugging.

## Adding a New Feature

Follow these steps to add a new feature package (e.g., `projects`):

### 1. Create the Package

```bash
cd packages/features
flutter create --template=package projects_content
```

### 2. Register in Workspace

Add the package to the root `pubspec.yaml`:
```yaml
workspace:
  - packages/features/projects_content
  # ... existing entries
```

### 3. Set Up pubspec.yaml

In `packages/features/projects_content/pubspec.yaml`:
```yaml
name: projects_content
resolution: workspace
environment:
  sdk: ^3.10.7

dependencies:
  auto_route: ^11.1.0
  flutter:
    sdk: flutter
  flutter_bloc: ^9.1.1
  get_it: ^9.2.0
  injectable: ^2.7.1+4
  shared: ^0.0.1

dev_dependencies:
  auto_route_generator: ^10.4.0
  build_runner: ^2.10.5
  injectable_generator: ^2.12.0
  very_good_analysis: ^10.0.0
```

### 4. Create the Feature Router

`lib/router/projects_content_router.dart`:
```dart
import 'package:auto_route/auto_route.dart';
import 'package:projects_content/presentation/pages/projects/projects_page.dart';

part 'projects_content_router.gr.dart';

@AutoRouterConfig(replaceInRouteName: 'Page,Route')
class ProjectsContentRouter extends RootStackRouter {
  @override
  List<AutoRoute> get routes => [
    AutoRoute(page: ProjectsRoute.page),
  ];
}
```

### 5. Create a Page

`lib/presentation/pages/projects/projects_page.dart`:
```dart
import 'package:auto_route/auto_route.dart';
import 'package:flutter/material.dart';

@RoutePage()
class ProjectsPage extends StatelessWidget {
  const ProjectsPage({super.key});

  @override
  Widget build(BuildContext context) {
    return const Scaffold(
      body: Center(child: Text('Projects')),
    );
  }
}
```

### 6. Set Up DI

`lib/di/injection.dart`:
```dart
import 'package:get_it/get_it.dart';
import 'package:injectable/injectable.dart';
import 'package:projects_content/di/injection.config.dart';

@InjectableInit(preferRelativeImports: true)
Future<void> configureProjectsContentDependencies(
  GetIt getIt, {
  required String environment,
}) async {
  getIt.init(environment: environment);
}
```

### 7. Export Public API

`lib/projects_content.dart`:
```dart
export 'di/injection.dart';
export 'presentation/pages/projects/projects_page.dart';
export 'router/projects_content_router.dart';
export 'router/projects_content_router.gr.dart';
```

### 8. Run Code Generation

```bash
cd packages/features/projects_content
flutter pub run build_runner build --delete-conflicting-outputs
```

### 9. Wire into the App

In `apps/cv_app/pubspec.yaml`, add the dependency:
```yaml
dependencies:
  projects_content: ^0.0.1
```

In `apps/cv_app/lib/di/injection.dart`, register the DI module:
```dart
import 'package:projects_content/di/injection.dart' as projects_content;

// Inside configureDependencies():
await projects_content.configureProjectsContentDependencies(
  getIt,
  environment: environment,
);
```

In `apps/cv_app/lib/router/router.dart`, add the route:
```dart
import 'package:projects_content/projects_content.dart';

@override
List<AutoRoute> get routes => [
  AutoRoute(page: HomeRoute.page, initial: true),
  AutoRoute(page: ProjectsRoute.page, path: '/projects'),
];
```

## Localization

Localization is centralized in the `shared` package.

- **Source ARB files**: `packages/core/shared/l10n/app_en.arb`
- **Config**: `packages/core/shared/l10n.yaml`
- **Generated output**: `packages/core/shared/lib/l10n/app_localizations.dart`

To add a new string, edit `app_en.arb` and run:
```bash
melos l10n:shared
```

To add a new locale, create `app_<locale>.arb` in the same directory and re-run generation.

Usage in widgets:
```dart
import 'package:shared/l10n/l10n.dart';

// In a build method:
AppLocalizations.of(context).appTitle
```

## Theming

Theme configuration lives in `packages/core/shared/lib/theme/theme.dart`. The `AppTheme` class provides both `light` and `dark` variants.

Colors are defined in `assets/colors/colors.xml` inside the `shared` package and auto-generated to `lib/gen/colors.gen.dart` by FlutterGen. Reference them via `ColorName.primaryColor`, `ColorName.secondaryColor`, etc.

The app widget applies the theme:
```dart
MaterialApp.router(
  theme: AppTheme.light,
  darkTheme: AppTheme.dark,
  // ...
)
```

## VS Code Configuration

The repository includes pre-configured VS Code settings:

- **`.vscode/settings.json`**: Points the Dart SDK to the FVM-managed Flutter installation (`.fvm/versions/stable`).
- **`.vscode/launch.json`**: Launch configurations for every app and package, including debug, profile, and release modes. The `cv_app` profile mode config uses `lib/main_dev.dart` as the program entry point.
