# Firebase Setup

## Projects

| Environment | Firebase Project ID | Entry Point |
|-------------|-------------------|-------------|
| Dev | `cv-web-app-dev` | `main_dev.dart` |
| Prod | `cv-web-app-prod` | `main_prod.dart` |

Firebase options files are generated by FlutterFire CLI:
- `apps/cv_app/lib/config/dev/firebase_options_dev.dart`
- `apps/cv_app/lib/config/prod/firebase_options_prod.dart`
- `apps/admin_app/lib/config/dev/firebase_options_dev.dart`
- `apps/admin_app/lib/config/prod/firebase_options_prod.dart`

Currently configured for **web only**.

## Firebase Services

| Service | Usage |
|---------|-------|
| **Firestore** | Document storage for profile, projects, work experience |
| **Auth** | Admin app authentication (email/password) |
| **Hosting** | Static web hosting for both apps (SPA rewrites) |
| **Analytics** | Event tracking |

## Hosting

Each app has its own `firebase.json` with SPA rewrites. Hosting sites:

| App | Dev Site | Prod Site |
|-----|----------|-----------|
| cv_app | `cv-web-app-dev` | `cv-web-app-prod` |
| admin_app | `cv-admin-app-dev` | `cv-admin-app-prod` |

The CV app deploys to the default hosting site (matching the project ID), so no extra config is needed. The admin app uses a **deploy target** (`admin`) configured in `apps/admin_app/.firebaserc` to resolve the correct site per environment. GitHub Actions workflows pass `target: admin` in the admin deploy step.

## Regenerating Firebase Config

```bash
cd apps/cv_app  # or apps/admin_app

# Dev
flutterfire configure \
  --project=cv-web-app-dev \
  --out=lib/config/dev/firebase_options_dev.dart \
  --platforms=web

# Prod
flutterfire configure \
  --project=cv-web-app-prod \
  --out=lib/config/prod/firebase_options_prod.dart \
  --platforms=web
```

## SEO / Crawlers

Both sites include `<meta name="robots" content="noindex, nofollow">`. The CV app also has a `robots.txt` disallowing all crawlers.
